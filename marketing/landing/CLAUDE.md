# Landing — Инструкции для работы с лендингами Unilist

## Роль
Ты работаешь над **текстами** лендинг-страниц продукта Unilist. Здесь не пишется код. Основная работа — создание, структурирование и улучшение текстов для блоков лендинга: заголовки, подзаголовки, описания, CTA, возражения, роли. Лендинги затем публикуются на Tilda.

## При старте сессии
1. Читай `../DOMAIN_STATE.md` — статус маркетинговых гипотез
2. Читай `../segments.md` — сегменты и JTBD
3. Читай `landing-sales-b2b.md` — структура и тексты основного лендинга (B2B + Bitrix24)
4. Читай `landing-problem-block.md` — блок «Проблема», тексты по ролям (РОП, Менеджер, Маркетолог, Производство)
5. Читай `../gtm-strategy.md` — GTM-стратегия, фазы роста, каналы

## Файлы в этой папке
- `landing-template.md` — **шаблон лендинга под сегмент (JTBD-структура)**. Использовать как базу при создании нового лендинга под любой сегмент.
- `landing-sales-b2b.md` — полная структура и тексты лендинга под Сегмент 1 (отделы продаж + Bitrix24)
- `landing-problem-block.md` — тексты блоков: Проблема, Как работает (5 табов), Роли (РОП / Менеджер / Маркетолог / Производство)
- `screenshots/` — скриншоты лендингов (реальное состояние, референсы, замечания)
- `CLAUDE.md` — этот файл

## Принципы работы с текстами лендинга
- Никаких абстракций. Только конкретика: "ответы в карточке Bitrix24", не "эффективная работа"
- Заголовки — как удар: проблема или выгода без вступлений
- CTA — всегда действие: "Создайте форму за 10 минут", не "Попробовать"
- Тон: прямой, уважительный, без маркетинговых штампов
- **Обращение — на "вы" с маленькой буквы** во всех лендингах. Уважительно, но без пафоса. "вы отправили анкету", не "Вы отправили анкету" и не "ты отправил анкету".
- Целевой читатель: РОП / собственник B2B-компании 20-200 человек

## Инструменты публикации (когда нужно опубликовать готовые тексты)
- Tilda: используй MCP-инструменты `mcp__tilda__*` для публикации и редактирования блоков
- Основная работа — текстовые файлы в этой папке и в `../`, а не код

## Связь с другими доменами
- Изменение позиционирования → обновить `../DOMAIN_STATE.md` + `../../CENTRAL_STATE.md`
- Новый сегмент/гипотеза → добавить в `../segments.md`
- Готовый кейс/результат → передать в `../../content/` для статьи

## Что НЕ делать
- Не создавать дублирующие файлы с текстами — основные тексты хранятся в `landing-*.md` внутри этой папки
- Не менять tone of voice без согласования (см. принципы выше)
- Не публиковать на Tilda без проверки текстов по принципам выше
