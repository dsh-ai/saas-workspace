# Фича: Участники опросника (Survey Participants)

**Дата:** 2026-03-26
**Статус:** Design approved, not implemented

---

## Проблема

Опросник отправляется сейлзом клиенту, клиент пересылает его в тендерный отдел. В сделке появляется новый участник — но владелец опросника узнаёт об этом на финальных переговорах, когда повлиять уже невозможно.

Текущее состояние: система не имеет понятия "участник". Есть только первичный контакт (`clientName`/`clientEmail`) на Survey-сущности. Кто реально заполняет форму — неизвестно.

---

## Два сценария отправки

**Сценарий 1 — Прямая отправка:** сейлз создаёт опросник, вручную указывает ФИО и компанию клиента, получает уникальную ссылку `/form/{uuid}`. Ссылка может быть переслана дальше по цепочке.

**Сценарий 2 — Публичная форма:** шаблон опубликован на сайте с общей ссылкой. Любой может открыть. Контактные данные собираются через обязательные поля в начале формы.

---

## Решение

### 1. Определение нового участника через sessionToken

При первом открытии формы в браузере генерируется `sessionToken` и сохраняется в `localStorage`. При каждом запросе к API токен передаётся в заголовке.

**Логика срабатывания интерстишла:**

- **Сценарий 1, первое открытие** — токена нет, но Survey содержит `clientName`. Молча создаём первого участника как `clientName`/`clientEmail`, сохраняем токен. Ничего не спрашиваем.
- **Сценарий 1, повторное открытие с нового устройства** — токена нет, но участник с `isPrimary: true` уже существует. Показываем интерстишл: "Вы [clientName]?" / "Нет, я другой человек".
- **Сценарий 2** — идентификация через обязательные контактные поля в начале формы (уже реализовано через `UpdateContactDto`).

**Интерстишл "Новый участник"** запрашивает:
- Имя (обязательно)
- Должность (опционально)
- Email (опционально — "чтобы мы могли вам ответить")

После заполнения — создаётся `SurveyParticipant`, владелец получает уведомление.

---

### 2. Делегирование

Кнопка "Делегировать коллеге" в форме заполнения. Не требует заполнения данных делегирующим — просто копирует текущую ссылку.

Когда делегат открывает ссылку — у него нет `sessionToken` → срабатывает интерстишл (см. выше).

**Режим доступа (зависит от тарифа):**

| Тариф | Режим | Поведение |
|-------|-------|-----------|
| Base | Последовательный | Только один участник редактирует в каждый момент. Кнопка "Передать ход". |
| Pro | Одновременный | Все участники редактируют параллельно (как Google Docs). Требует real-time sync. |

---

### 3. Новая сущность: SurveyParticipant

```
SurveyParticipant {
  id: uuid
  surveyId: string          // FK → Survey
  sessionToken: string      // уникальный токен сессии
  name: string              // введено в интерстишле
  role: string | null       // должность
  email: string | null      // опционально
  isPrimary: boolean        // true = первичный контакт из CRM
  status: 'viewed' | 'in_progress' | 'completed'
  isActiveEditor: boolean   // для режима последовательного редактирования
  createdAt: Date
  updatedAt: Date
}
```

---

### 4. Отображение в дашборде

В карточке опросника — раздел "Участники":

```
● Иван Петров (первичный контакт)        ✓ заполнил
● Анна Соколова · Начальник тендерного   ⏳ в процессе
● [неизвестно]                           👁 просмотрел
```

Уведомление владельцу при появлении нового участника — в дашборде + опционально email.

---

### 5. Синхронизация с Bitrix24

При создании нового `SurveyParticipant` (не первичного):

1. **Создать контакт** `crm.contact.add` — Имя, Должность, Email (если есть)
2. **Привязать к сделке** `crm.deal.contact.add` — к `bitrixEntityId` из Survey
3. **Запись в таймлайн сделки** через существующий `sendBitrixTimelineComment()`: "Новый участник в опроснике: Анна Соколова, Начальник тендерного отдела"

Если `bitrixEntityId` не задан (публичная форма) — контакт создаётся без привязки к сделке. Владелец привязывает вручную из дашборда.

---

## Что не входит в скоуп (сейчас)

- Одновременное редактирование (Pro-тариф) — отдельная задача, требует WebSocket real-time sync
- Автоматическое создание сделки для публичных форм
- Уведомления в Slack/Telegram
